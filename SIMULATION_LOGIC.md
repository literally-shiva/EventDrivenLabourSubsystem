# Улучшенная логика генерации данных симуляции

## Обзор изменений

Полностью переработана логика генерации метрик в `SimulationBackgroundService` для создания реалистичных данных с характерными паттернами событий.

## Ключевые улучшения

### 1. Накопительные метрики

**Раньше:** Каждый тик генерировал случайные значения независимо от истории.

**Теперь:** Метрики накапливаются со временем:
- **Collision Count** — растёт постепенно, скачки при событии `CollisionDetected`
- **Changes Count** — накапливается при изменениях ТЗ
- **Rework Count** — увеличивается при возвратах на доработку
- **Documentation Versions** — растёт естественным образом + скачки при согласованиях

### 2. Фазы проекта

Проект разделён на 4 фазы в зависимости от прогресса (0-100%):

| Фаза | Прогресс | Характеристики |
|------|----------|----------------|
| **Initial** (Начальная) | 0-20% | Мало коллизий, возможны изменения ТЗ, нехватка ресурсов |
| **Development** (Разработка) | 20-50% | Пик коллизий, активные изменения, высокая вероятность событий |
| **Integration** (Интеграция) | 50-80% | Много согласований, задержки экспертизы, возвраты на доработку |
| **Completion** (Завершение) | 80-100% | Доработки документации, финальные согласования |

### 3. Характерные паттерны событий

Каждое событие теперь имеет **уникальную сигнатуру** изменения метрик:

#### CollisionDetected (Обнаружена коллизия)
```
✓ Скачок коллизий: +3-7
✓ Замедление прогресса: 2-7% вместо 8-20%
✓ Вероятность: выше в фазе Development (40%)
```

#### DesignRequirementChanged (Изменение ТЗ)
```
✓ Скачок изменений ТЗ: +2-5
✓ Дополнительные документы: +1-2
✓ Замедление прогресса
✓ Вероятность: высокая в Initial и Development
```

#### DocumentationReturned / ExpertReviewFailed (Возврат на доработку)
```
✓ Скачок возвратов: +2-4
✓ Больше версий документов: +2-4
✓ Сильное замедление прогресса
✓ Вероятность: высокая в Integration и Completion (40-70%)
```

#### ApprovalDelayed (Задержка согласования)
```
✓ Накопление документов на согласовании: +3-6
✓ Задержка: 5-12 дней (вместо обычных 0-2)
✓ Дополнительные согласования: +1-2
✓ Вероятность: пик в Integration (30%)
```

#### ResourceShortage (Нехватка ресурсов)
```
✓ Временное снижение работников: -2-3 от базового
✓ Небольшой рост задержек
✓ Вероятность: выше в начале проекта
```

### 4. Естественный рост метрик

Даже без событий метрики меняются естественным образом:

- **Коллизии** — постепенно растут в фазе разработки (30% шанс +0-1 каждый тик)
- **Документы** — накапливаются (20% шанс +1 каждый тик, макс 8)
- **Работники** — небольшие колебания ±1 от базового значения
- **Объём модели** — линейный рост с прогрессом: 150 → 600 MB

### 5. Ограничения роста

Метрики имеют разумные верхние границы:

```csharp
Collision Count max: 15
Changes Count max: 12
Rework Count max: 10
Approval Count max: 8
Documentation Versions max: 12
Workers Count: 2-10
```

### 6. Интервалы между событиями

**Частота тиков:** 1 секунда (каждую секунду генерируется новый снимок метрик)

**Минимальный интервал между событиями:** 2 дня

**Вероятность события зависит от фазы:**
- Initial: 20%
- Development: 30% (пик активности)
- Integration: 35% (максимум событий)
- Completion: 25%

События кластеризуются в проблемных фазах (Development, Integration), создавая визуально различимые паттерны на графике.

## Примеры реалистичных сценариев

### Сценарий 1: Проблемная разработка

```
День 1-5: Норма, коллизии = 1-2
День 6: CollisionDetected → коллизии = 7
День 7-10: Норма, коллизии = 7-9 (естественный рост)
День 11: DesignRequirementChanged → изменения ТЗ = 4, документы +2
День 12-15: Норма
День 16: CollisionDetected → коллизии = 15 (достигнут лимит)
```

График коллизий будет показывать:
- Базовый уровень 1-2
- Скачок до 7 (событие)
- Плавный рост 7→9
- Скачок до 15 (событие)
- Плато на уровне 15

### Сценарий 2: Задержки согласования

```
День 1-10: Норма, документы = 2-4
День 11: ApprovalDelayed → документы = 8, задержка = 7 дней
День 12-14: Документы на уровне 8-9
День 15: DocumentationReturned → возвраты = 3, документы = 12
День 16-20: Доработки, возвраты = 3-4
```

График документов покажет:
- Стабильный низкий уровень
- Резкий скачок при задержке
- Дополнительный скачок при возврате
- Постепенное снижение после доработок

## Технические детали

### Состояние работы

Каждая работа хранит свое состояние в памяти:

```csharp
private class WorkSimulationState
{
    public int BaseWorkersCount { get; set; }      // Базовое количество работников
    public int BaseCollisions { get; set; }        // Накопленные коллизии
    public int BaseChanges { get; set; }           // Накопленные изменения ТЗ
    public int BaseRework { get; set; }            // Накопленные возвраты
    public int BaseApprovals { get; set; }         // Количество согласований
    public int BaseDocVersions { get; set; }       // Версии документов
    public int DaysSinceLastEvent { get; set; }    // Дней с последнего события
}
```

### Алгоритм генерации события

```csharp
1. Проверка: прошло ли 2+ дней с последнего события?
   НЕТ → SimulatedEventType.None
   ДА → продолжить

2. Определить фазу проекта по прогрессу (0-20%, 20-50%, 50-80%, 80-100%)

3. Рассчитать вероятность события для фазы (20-35%)

4. Если событие произошло → выбрать тип в зависимости от фазы:
   - Initial: 30% ResourceShortage, 30% DesignChanged, 20% ApprovalDelayed
   - Development: 40% CollisionDetected, 30% DesignChanged
   - Integration: 30% ApprovalDelayed, 30% DocumentationReturned, 20% ExpertFailed
   - Completion: 40% DocumentationReturned, 30% ExpertFailed

5. Обновить накопительные метрики согласно событию
```

## Визуальные паттерны на графике

### Нормальный проект
```
Коллизии:     ─────┐        ┌──────
                   └────────┘
Изменения:    ───┐     ┐──────────
                 └─────┘
Возвраты:     ────────────┐  ┌────
                          └──┘
```
Плавные линии, редкие скачки, быстрое восстановление.

### Проблемный проект
```
Коллизии:     ─┐  ┌─┐    ┌──┐  ┌──
               └──┘ └────┘  └──┘
Изменения:    ─┐ ┌┐  ┌─┐  ┌───┐
               └─┘└──┘ └──┘   └──
Возвраты:     ───┐ ┌──┐ ┌───┐
                  └─┘  └─┘   └────
```
Частые скачки, высокие значения, медленное восстановление.

## Сравнение старой и новой логики

| Аспект | Старая логика | Новая логика |
|--------|---------------|--------------|
| Накопление | Нет, каждый тик случаен | Да, метрики накапливаются |
| События | Случайные 23% времени | Зависят от фазы, 15-30%, минимум 3 дня между |
| Паттерны | Нет характерных сигнатур | Уникальные скачки для каждого события |
| Тренды | Нет | Коллизии растут в Development, согласования в Integration |
| Реализм | Хаотичный шум | Структурированные паттерны |
| График | Неразличимые линии | Чёткие паттерны, видимые события |

## Тестирование

### Проверка накопления

1. Запустите симуляцию
2. Следите за графиком Collision Count
3. При событии `CollisionDetected` должен быть скачок +3-7
4. Между событиями значение должно плавно расти или оставаться стабильным

### Проверка фаз

1. Отслеживайте тип событий в начале проекта (0-20%)
2. Должны преобладать: ResourceShortage, DesignRequirementChanged
3. В середине (20-50%) должны появиться частые CollisionDetected
4. Ближе к концу (80-100%) — DocumentationReturned, ExpertReviewFailed

### Проверка интервалов

1. Зафиксируйте событие на графике (маркер)
2. Следующее событие для этой работы должно быть минимум через 2 дня
3. С учётом частоты тиков 1 секунда = 1 день, события появляются примерно каждые 2-10 секунд реального времени

## Настройка параметров

Основные параметры находятся в коде и могут быть изменены:

```csharp
// Частота генерации данных
await Task.Delay(TimeSpan.FromSeconds(1), ...);  // 1 секунда

// Вероятность событий по фазам
ProjectPhase.Initial => 0.20,      // 20%
ProjectPhase.Development => 0.30,  // 30%
ProjectPhase.Integration => 0.35,  // 35%
ProjectPhase.Completion => 0.25,   // 25%

// Минимальный интервал между событиями
if (state.DaysSinceLastEvent < 2)  // 2 дня

// Границы роста метрик
state.BaseCollisions = Math.Min(15, ...);
state.BaseChanges = Math.Min(12, ...);
// и т.д.
```

## Возможные расширения

### 1. Корреляции между работами

Если в одной работе много коллизий, увеличить вероятность коллизий в зависимых работах.

### 2. Сезонность

Меньше событий в начале и конце недели, больше в середине.

### 3. Персистентное состояние

Сохранять `WorkSimulationState` в БД для сохранения между перезапусками.

### 4. Настраиваемые профили

"Стабильный проект", "Проблемный проект", "Кризисный проект" с разными вероятностями.

## Заключение

Новая логика генерации создаёт **структурированные, реалистичные данные** вместо хаотичного шума. Каждое событие теперь имеет характерную сигнатуру, которую можно распознать на графике, что делает систему пригодной для демонстрации и обучения ML-моделей.
