# Улучшенные паттерны событий — Итоговый отчёт

## Дата реализации: 2026-06-06

## Контекст изменений

Предыдущая реализация генерации паттернов событий имела критические недостатки:
1. **Отсутствие каскадных эффектов** — события изменяли только "свои" метрики
2. **Идентичная обработка разных событий** — ExpertReviewFailed = DocumentationReturned
3. **Семантические ошибки** — ApprovalDelayed увеличивал версии документов
4. **Несоответствие весам** — ExpertReviewFailed (×1.20) обрабатывался как DocumentationReturned (×1.15)

Это делало события **неразличимыми на графике** и непригодными для обучения ML-модели.

## Технические детали реализации

### Новое поле WorkSimulationState

Добавлено поле для хранения временных задержек от событий:

```csharp
public int TemporaryApprovalDelay { get; set; }
```

Это поле используется для передачи задержек от событий (DocumentationReturned, ExpertReviewFailed, ResourceShortage) в метод `GenerateApprovalDelay`.

### Обновлённый метод GenerateApprovalDelay

```csharp
private static int GenerateApprovalDelay(WorkSimulationState state, SimulatedEventType eventType)
{
    // 1. Проверяем событие текущего тика
    if (eventType == SimulatedEventType.ApprovalDelayed)
    {
        return Random.Shared.Next(5, 13);
    }

    // 2. Проверяем временную задержку от предыдущих событий
    if (state.TemporaryApprovalDelay > 0)
    {
        var delay = state.TemporaryApprovalDelay;
        state.TemporaryApprovalDelay = 0;  // сбрасываем
        return delay;
    }

    // 3. Обычные задержки
    return Random.Shared.Next(0, 3);
}
```

Теперь задержки от событий DocumentationReturned (+2-4), ExpertReviewFailed (+7-12), ResourceShortage (+2-5) корректно отображаются в метрике ApprovalDelayDays.

## Итоговое резюме

Добавлены **каскадные эффекты** для всех событий согласно физической логике BIM-процессов. ExpertReviewFailed **усилен в 1.5-2 раза**. Исправлены **семантические ошибки** в ApprovalDelayed и ResourceShortage.
