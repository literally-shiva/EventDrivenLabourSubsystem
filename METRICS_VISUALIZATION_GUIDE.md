# Руководство по визуализации метрик

## Что реализовано

Добавлена новая вкладка "Metrics" в клиентское приложение для отслеживания показателей цифрового двойника в реальном времени.

**Важно:** Логика генерации данных полностью переработана для создания реалистичных паттернов вместо хаотичного шума. Подробности в [SIMULATION_LOGIC.md](SIMULATION_LOGIC.md).

## Компоненты изменений

### Backend (CoreServer)

1. **IProjectQueryService** - добавлен метод `GetMetricsAsync`
2. **MetricHistoryDto** - новый DTO для передачи метрик клиенту
3. **ProjectQueryService** - реализация получения истории метрик из БД
4. **ProjectsController** - новый эндпоинт `GET /api/projects/{id}/metrics`

### Frontend (Angular)

1. **models.ts** - интерфейс `MetricHistory`
2. **api.service.ts** - метод `getMetrics(projectId)`
3. **app.component.html** - вкладки (Gantt / Metrics) и canvas для графика
4. **app.component.ts** - логика отрисовки графика с маркерами событий
5. **app.component.css** - стили для вкладок и графика

## Визуализируемые метрики

График отображает следующие показатели проектного процесса:

1. **Collision Count** (красный) — количество коллизий в BIM-модели
2. **Changes Count** (оранжевый) — количество изменений технического задания
3. **Rework Count** (фиолетовый) — количество возвратов на доработку
4. **Approval Delays** (синий) — задержки получения технических условий (дни)
5. **Documentation Versions** (зелёный) — количество документов на согласовании
6. **Approval Count** (розовый) — количество согласований

### Соответствие требуемым показателям

Из требуемых 9 показателей реализованы 6 на основе доступных данных в `MetricHistory`:

✅ Количество коллизий → `collisionCount`  
✅ Количество изменений ТЗ → `changesCount`  
✅ Количество возвратов на доработку → `reworkCount`  
✅ Задержки получения ТУ → `approvalDelayDays`  
✅ Документы на согласовании → `documentationVersionCount`  
✅ Согласования → `approvalCount`

**Отсутствуют в модели данных:**
- ❌ Количество замечаний экспертизы (частично покрывается `reworkCount`)
- ❌ Количество несогласованных решений
- ❌ Количество изменений BIM-модели (есть `modelDataVolume`, но это объём, а не счётчик изменений)
- ❌ Количество нормативных несоответствий

Для добавления отсутствующих показателей необходимо расширить сущность `MetricHistory` в CoreServer и DigitalTwin.

## Маркеры событий

События отображаются как:
- Цветные точки на оси времени (внизу графика)
- Вертикальные пунктирные линии через весь график

Цвета событий:
- `ApprovalDelayed` - синий
- `CollisionDetected` - красный
- `DesignRequirementChanged` - фиолетовый
- `DocumentationReturned` - оранжевый
- `ResourceShortage` - розовый
- `ExpertReviewFailed` - оранжевый-красный

## Как использовать

1. Запустите CoreServer, MLService, DigitalTwin
2. Откройте ProjectClient в браузере
3. Создайте или откройте проект
4. Запустите симуляцию
5. **Подождите 1-2 минуты** для накопления данных
6. Переключитесь на вкладку **Metrics**
7. График автоматически загрузит историю метрик и отобразит их

**Что ожидать:**
- Первые 3-5 тиков: базовые значения метрик (коллизии 0-2, изменения 0-1)
- Затем: постепенный рост накопительных метрик
- При событиях: характерные скачки (коллизии +3-7, изменения ТЗ +2-5)
- Маркеры событий внизу графика показывают моменты скачков

**Примечание:** Новая логика генерирует структурированные данные с реалистичными паттернами. Каждое событие оставляет характерную сигнатуру на графике.

## График обновляется

- При переключении на вкладку Metrics
- При смене активного проекта (необходимо переключиться на вкладку заново)

## Технические детали реализации

### Проблема с *ngIf и ViewChild

Gantt-диаграмма теперь условно отображается через `*ngIf="activeTab === 'gantt'"`, что создаёт проблему с инициализацией:

**Решение:**
1. `@ViewChild` изменён на `{ static: false }` — элемент доступен после рендеринга
2. При каждом переключении на вкладку Gantt вызывается `gantt.init()` заново
3. Используется `setTimeout(100ms)` для ожидания рендеринга DOM
4. Метод `switchToTab()` координирует переключение и переинициализацию

Это гарантирует, что Gantt всегда корректно привязан к DOM-элементу после переключения вкладок.

## Архитектурные решения

### Почему Canvas API, а не библиотека?

- Легковесное решение без дополнительных зависимостей
- Полный контроль над отрисовкой
- Высокая производительность для большого количества точек
- Простота кастомизации под требования проекта

### Scaling и DPI

График автоматически адаптируется к DPI экрана (retina displays), используя `devicePixelRatio`.

### Временная ось

- Автоматически масштабируется по всему диапазону данных
- Отображает дату в формате DD.MM и время HH:MM на двух строках
- Показывает до 6 меток времени для читаемости

### Ось Y

- Автоматически масштабируется по максимальному значению из всех метрик
- 5 горизонтальных линий-направляющих
- Значения округлены до целых чисел

## Известные ограничения

1. **Нет автообновления в реальном времени** - график загружается только при переключении на вкладку. Для обновления нужно переключиться на другую вкладку и вернуться.

2. **Нет зума и панорамирования** - график статичен, отображает все данные сразу.

3. **Нет интерактивных подсказок** - при наведении на точку не отображается значение.

4. **Производительность при большом объёме данных** - при длительной симуляции (1000+ точек) возможна задержка отрисовки.

## Связь с архитектурой

Новая функция **не нарушает** существующую архитектуру:

- Использует существующий `IMetricRepository.GetByProjectAsync`
- Добавляет только read-only эндпоинт
- Не влияет на алгоритмы DBSCAN/SVM/Markov
- Не изменяет поток данных DigitalTwin → CoreServer

Это **чисто визуализационный слой**, параллельный существующему Gantt-представлению.
