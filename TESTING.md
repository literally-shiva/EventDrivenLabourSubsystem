# Инструкция по тестированию системы

## Предварительные требования

1. PostgreSQL запущен на порту 5432
2. Все зависимости установлены:
   - .NET 10.0 SDK
   - Python 3.10+ с venv
   - Node.js 18+ и npm

## Быстрый запуск

### Вариант 1: Автоматический запуск всех сервисов

```bash
./start-all.sh
```

Этот скрипт запустит все 4 сервиса в отдельных терминалах в правильном порядке.

### Вариант 2: Ручной запуск каждого сервиса

Откройте 4 терминала и выполните в каждом:

**Терминал 1 - MLService:**
```bash
./start-mlservice.sh
```

**Терминал 2 - CoreServer:**
```bash
./start-coreserver.sh
```

**Терминал 3 - DigitalTwin:**
```bash
./start-digitaltwin.sh
```

**Терминал 4 - ProjectClient:**
```bash
./start-projectclient.sh
```

Откройте браузер: `http://localhost:4200`

Должна открыться страница с интерфейсом управления проектами.

## Тестовый сценарий

### Сценарий 1: Создание проекта и запуск симуляции

1. Откройте `http://localhost:4200`
2. Нажмите "Create New Project"
3. Добавьте 3 работы кнопкой "Add Work"
4. Настройте работы:
   - Work 1: 5 дней
   - Work 2: 7 дней (зависит от Work 1)
   - Work 3: 6 дней (зависит от Work 1)
5. Создайте зависимости в Gantt диаграмме (перетащите линии)
6. Нажмите "Save Project"
7. Нажмите "Start Simulation"
8. Наблюдайте за обновлениями в реальном времени

### Сценарий 2: Обработка событий

После запуска симуляции:

1. DigitalTwin генерирует метрики каждые 3 секунды
2. Метрики отправляются в CoreServer
3. CoreServer вызывает MLService для кластеризации
4. Обнаруженные события классифицируются через SVM
5. Неизвестные события показываются в модальном окне
6. Пользователь может дать название новому событию
7. Состояния работ обновляются через Марковские цепи
8. Длительность работ пересчитывается
9. Gantt диаграмма обновляется через SignalR

### Сценарий 3: Экспорт/Импорт проекта

1. Создайте проект с несколькими работами
2. Нажмите "Export JSON"
3. Сохраните файл
4. Создайте новый проект
5. Нажмите "Import JSON" и выберите сохраненный файл
6. Проект должен загрузиться со всеми работами и зависимостями

## Типы событий

Система генерирует следующие типы событий:

- **DesignRequirementChanged** (2% вероятность) - изменение требований
- **DocumentationReturned** (3% вероятность) - возврат документации
- **CollisionDetected** (4% вероятность) - обнаружение коллизий
- **ApprovalDelayed** (4% вероятность) - задержка согласования
- **ResourceShortage** (8% вероятность) - нехватка ресурсов
- **ExpertReviewFailed** (2% вероятность) - неудачная экспертиза
- **None** (77% вероятность) - штатная работа

## Состояния работ (Марковские цепи)

- **S0_Stable** - стабильное состояние
- **S1_LowSensitive** - низкая чувствительность к событиям
- **S2_MediumSensitive** - средняя чувствительность
- **S3_HighSensitive** - высокая чувствительность
- **S4_Critical** - критическое состояние

## Troubleshooting

### Ошибка: "Cannot connect to PostgreSQL"
```bash
# Проверьте статус PostgreSQL
systemctl status postgresql

# Проверьте порт
ss -tuln | grep 5432
```

### Ошибка: "Port already in use"
```bash
# Найдите процесс на порту
lsof -i :5000  # или другой порт

# Убейте процесс
kill -9 <PID>
```

### Ошибка: "MLService not responding"
```bash
# Проверьте что venv активирован
source solution/MLService/.venv/bin/activate

# Переустановите зависимости
pip install -r solution/MLService/requirements.txt
```

### Ошибка: "Angular compilation failed"
```bash
cd solution/ProjectClient
rm -rf node_modules package-lock.json
npm install
```

## Ожидаемое поведение

1. **При создании проекта**: проект сохраняется в DigitalTwin БД
2. **При запуске симуляции**: 
   - Каждые 3 секунды генерируются метрики
   - Метрики отправляются в CoreServer
   - События обрабатываются через ML
   - UI обновляется через SignalR
3. **При обнаружении события**:
   - Событие классифицируется (известное/неизвестное)
   - Состояние работы обновляется
   - Длительность пересчитывается
   - Gantt обновляется
4. **При остановке симуляции**: генерация метрик прекращается

## Метрики производительности

- **Частота генерации метрик**: каждые 3 секунды
- **Время обработки события**: < 1 секунда
- **Задержка SignalR**: < 100ms
- **Время кластеризации (DBSCAN)**: < 50ms
- **Время классификации (SVM)**: < 10ms
